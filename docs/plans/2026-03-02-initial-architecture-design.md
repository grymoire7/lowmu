# lowmu: Initial Architecture Design

**Date:** 2026-03-02
**Status:** Approved

## Overview

lowmu is a Ruby CLI tool that lowers the friction of publishing blog posts and
related content to the social web. It takes a source markdown file and hero
image, uses AI to generate platform-specific content, and then publishes to
configured targets.

The workflow is split into two phases: **generation** (AI-assisted, writes
files to disk for review) and **publishing** (sends content to target
platforms). These are always separate steps so the user can review and edit
generated content before it goes anywhere.

## Architecture

**Approach: Thor + service objects**

Thor handles CLI argument parsing. Command classes are thin shells that validate
input and delegate immediately to service objects. Service objects carry all
business logic. This keeps the CLI interface decoupled from platform-specific
logic and makes every layer independently testable.

## Commands

### `lowmu new <md_path> <hero_image_path>`

1. Derives slug from the markdown filename (`my-new-post.md` → `my-new-post`)
2. Creates `content_dir/<slug>/`
3. Copies source `.md` → `original_content.md`, hero image → `hero_image.<ext>`
4. Writes initial `status.yml` with all targets set to `pending`

### `lowmu generate <slug> [--target <name>]`

1. Loads `original_content.md` and parses front matter
2. Reads `publish_to:` list to determine which generators to run; if `--target`
   is given, runs only that target (must be present in `publish_to:`)
3. For each selected target, instantiates the appropriate Generator and calls
   `generate(slug_dir, target_config)`
4. Each Generator uses RubyLLM to transform/rewrite content for that platform
   and writes the output file
5. Updates `status.yml`: `pending` → `generated` per target
6. Requires `--force` if the target already has `generated` status, to prevent
   overwriting user edits

### `lowmu publish <slug> [--target <name>]`

1. Reads `status.yml` — only processes targets with status `generated`; if
   `--target` is given, restricts to that single target
2. For each eligible target, instantiates the Publisher and calls
   `publish(slug_dir, target_config)`
3. Hugo: copies files to `base_path` in the Hugo content directory
4. Substack / Mastodon: call their respective APIs
5. LinkedIn: prints the path to `linkedin.md` for manual copy-paste; marks
   `published` with a note
6. Updates `status.yml`: `generated` → `published` with timestamp

The `--target` flag makes retrying a failed publish safe and precise — e.g.,
`lowmu publish my-post --target mastodon` retries only Mastodon without
re-publishing to other targets.

### `lowmu status [slug]`

- With slug: prints a target-by-target status table for that post
- Without slug: prints a summary row for every slug in `content_dir`

### `lowmu configure`

Writes a commented template config to `~/.config/lowmu/config.yml`. Prompts
before overwriting if a config already exists.

## Configuration

Located at `~/.config/lowmu/config.yml`.

```yaml
content_dir: ~/projects/lowmu/content

llm:
  provider: anthropic        # passed to RubyLLM
  model: claude-opus-4-6

targets:
  - name: tracyatteberry
    type: hugo
    base_url: https://tracyatteberry.com
    base_path: ~/projects/tracyatteberry/content

  - name: substack
    type: substack
    auth:
      type: api_key
      api_key: your_substack_api_key_here

  - name: mastodon
    type: mastodon
    base_url: https://mastodon.social
    auth:
      type: oauth
      access_token: your_access_token_here

  - name: linkedin
    type: linkedin             # generate-only, no auth needed
```

API credentials can alternatively be provided via environment variables
(e.g., `LOWMU_SUBSTACK_API_KEY`) as an alternative to storing secrets in the
config file.

## AI Content Generation

Content generation uses the **RubyLLM gem**, configured with the provider and
model from `config.yml`. Anthropic/Claude is the default for development.

Each Generator class is responsible for constructing its own prompt, appropriate
to its platform's format and constraints (e.g., Mastodon's character limit,
LinkedIn's professional tone, Hugo's markdown conventions).

## Error Handling

- Expected errors use a `Lowmu::Error` base class and are caught at the command
  layer, printed as clean user-facing messages with no backtrace.
- Unexpected errors surface normally. Set `LOWMU_DEBUG=1` to always show full
  backtraces.
- `status.yml` is the source of truth. If a step fails mid-run, the target
  status stays at its previous state so re-running is safe.
- `lowmu generate` requires `--force` to regenerate content for a target that
  is already in `generated` state, preventing accidental overwrites of edits.

## Testing Strategy

**Framework:** RSpec
**Style:** TDD red-green-refactor for all development

### Unit tests (fast, no I/O, no network)

- `Config` — valid YAML loading, raises on missing required fields
- `ContentStore` — slug derivation, directory/file layout
- Each `Generator` — given a fixture markdown file, produces correct output
  string; RubyLLM is doubled at the boundary
- Each `Publisher` — HTTP clients are doubled; correct payload assembly and
  method dispatch are asserted

### Integration tests (real file I/O, VCR for network)

- `Commands::New` — runs against a temp directory, asserts files and `status.yml`
- `Commands::Generate` — runs against a pre-populated temp slug dir, asserts
  output files and status updates; network calls captured by VCR
- `Commands::Status` — asserts correct output format

### Network stubbing

**VCR** (with **WebMock**) records and replays all HTTP interactions —
Anthropic/RubyLLM calls, Substack API, Mastodon API. Cassettes are committed
to git. VCR is configured with `filter_sensitive_data` to scrub secrets before
cassettes are written. To re-record against live APIs, delete the cassettes and
run with real credentials.

### Coverage

**SimpleCov** is configured in `spec/spec_helper.rb` and runs automatically
with every RSpec invocation. A minimum coverage threshold is enforced so
coverage regressions are caught early.

### Fixtures

Sample markdown files, hero images, and config YAMLs live in `spec/fixtures/`.
