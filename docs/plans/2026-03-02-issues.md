## Concerns and Issues

### 1. No distinction between "note" and "post" for input

Currently, the system does not differentiate between a "note" and a "post" when
processing input content. It makes a difference in how the content should be
processed. For example, when given a long-form "post" the mastodon generator
needs to create short-formt "note" for publishing. However, when given a
"note" the mastodon generator should preserve the short-form content for publishing
as long it is not too long for mastodon.

As another example, substack (and linkedin) have both "post" and "note" content
types. When given a "post" the substack generator should create both a "post"
and a "note" for publishing. However, when given a "note" the substack
generator should produce only a substack "note" and preserve the short-form
content for publishing as long it is not too long for substack. There would
also be a difference in API calls for creating a "post" vs a "note" on substack.

This may be a fundamental design issue thate we need to address.

### 2. Unimplemented Substack publisher

The substack publisher is currently unimplemented. This is a significant issue
and should be noted as unfinished work.

### 3. Hardcoded configuration in code

`lib/lowmu/commands/configure.rb` contains a default configuration file as a heredoc.
It would be better to have a separate default configuration file that can be easily
edited and maintained without modifying the code. For example, `default_config.yml`.

### 4. Model should not assume a default value in code

In `lib/lowmu/generators/base.rb`, the code currently assumes a default model:

```ruby
  model = @llm_config.fetch("model", "claude-opus-4-6")
```

We should not assume a default model/provider for the user. If one is not configured,
the system should raise an error and prompt the user to configure a model/provider
before using the system.

### 5. `lowmu tree` command

Where does the `lowmu tree` command come from that we see in `lowmu help`? It
does not seem to be implemented in the codebase and just produces a list of
lowmu commands which, since we have no nested subcommands, is not very useful.
Can we get rid of this?
